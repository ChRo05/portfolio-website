## 1. 背景・目的（Why）

業務効率化・AI・財務系のツールを複数開発してきたが、それらが散在していて第三者に実力が伝わらない。実績を1か所に集約し、案件獲得・キャリア形成につなげるための公開ポートフォリオを構築する。

## 2. ゴール（優先順位つき）

| 優先 | ゴール | このサイトでの役割 |
|---|---|---|
| 1 | フリーランス案件の獲得 | 「発注したら何を任せられるか」を即伝える |
| 2 | 転職活動 | 実装力・思考力の証明 |
| 3 | 名刺（恒常的な実績置き場） | URLひとつで自分を説明できる |

→ 最優先がフリーランスなので、サイト全体は「クライアント目線（＝この人に頼むと何が解決するか）」を主軸に設計する。

## 3. ターゲット（誰に見せるか）

両方に見せる構成（入口は易しく、奥は技術的に）。

- **ペルソナA：技術が分からない発注者**（中小企業の経営者・バックオフィス担当）
  知りたいこと：「うちの面倒な作業、自動化できる？」「実績ある？」「安心して任せられる？」
- **ペルソナB：技術が分かる人**（エンジニア・技術責任者・エージェント）
  知りたいこと：「コードの質」「技術スタック」「設計の考え方」

## 4. 強み と 一言キャッチ（確定）

コア：**「現場の業務を理解し、それを自動化・AIで"実装まで"持っていける人」**

3つの強みの重なりが特徴：

1. **業務ドメインの理解**（会計・財務・バックオフィス）
   ← asset-management-ledger / roi-simulator / report-hub / リース等の会計ツール
2. **自動化・RPAの実装力**（地に足のついた効率化）
   ← GAS / VBA で実業務をRPA化
3. **AI・LLMを実務に組み込む力**（今ホットな領域）
   ← ローカル完結LLMツール / ai-strategy-agent

### 一言キャッチ（確定＝合体案）

> **会計・バックオフィスの"面倒"を、AIと自動化で"実装まで"片付けるエンジニア**

- サブコピー候補（トップで補足する1行）：
  「業務を理解 → 自動化・AIで実装 → 成果まで。の3点セットで動けます。」

## 4.5 実績の見せ方（★今回の最重要論点・新規）

### 現状の問題

公開GitHub（ChRo05）に現在あるのは **学習用リポ＋fork のみ**で、4章で挙げた強みの本体（会計ツール・AI/LLMツール）は**1件も第三者から見えない**。これは1章の課題そのもの。

| 公開リポ | 種別 | ポートフォリオ価値 |
|---|---|---|
| python-certified-basic-jp | 学習教材 | △（学習姿勢のみ） |
| omikuji_experience / py | 学習ノート | △ |
| Tello-Python / node-tello-edu / samurai_express_sample | fork | ×（自作でない） |

→ **ポートフォリオの成否は「非公開の本体をどう見せるか」にかかっている。**

### 方針（採用＝ハイブリッド）

- **公開できる1〜2件**：守秘に触れない範囲でリポジトリを**公開化**し、コードを見せる（ペルソナBに最強）
- **機密案件**：コードは出さず、**スクショ＋デモ動画＋「課題→対応→成果」解説**でぼかして見せる（ペルソナA向け）
- 各実績カードに「公開コードあり／成果のみ」のバッジで見せ方を明示

## 5. スコープ（載せる/載せない）

**載せる（In）**
プロフィール / 強み / 制作実績（3〜5件）/ 技術スタック / note記事 / GitHubリンク / 連絡導線

**載せない（Out）— 守秘・個人情報**
- 機密案件の顧客名・具体的な業務内容・実データ（ぼかして表現）
- 履歴書・職務経歴書の個人情報（氏名以外。別途PDF限定共有）
- 住所・生年月日・電話番号
- **メールアドレス**（連絡はSNS/GitHub/note経由とする）

## 6. 機能要件

| ID | 機能 | 優先度 |
|---|---|---|
| F-1 | トップ：強みと一言キャッチを表示 | 高 |
| F-2 | 制作実績一覧（課題→対応→成果の型） | 高 |
| F-3 | 各実績の詳細（リンク・画像・「公開コード/成果のみ」バッジ） | 高 |
| F-4 | 技術スタック表示 | 中 |
| F-5 | note記事へのリンク | 中 |
| F-6 | 連絡導線（**SNS / GitHub / note リンク**。メール・フォームは無し） | 高 |
| F-7 | スマホ対応 | 高 |

## 7. 非機能要件

- 表示速度：軽量（画像最適化）
- レスポンシブ：スマホ・PC両対応
- 保守性：自分で更新できる構成
- **公開基盤：GitHub Pages（確定）**。フロント実装力のアピールを兼ねる。独自ドメインは任意。

## 8. 制約条件

- 機密保持：現職/受託案件の情報は出さない（note記事と同じぼかし基準）
- 個人情報保護：履歴書系は非公開
- 予算：原則無料枠（GitHub Pages）

## 9. 成功基準（KPI）— [要加筆]

- 定量（案・要確定）: 公開後3か月で **問い合わせ ◯件 / エージェント面談 ◯件**
- 定性: 「何ができる人か」が**30秒で伝わる**

> あなたが現実的だと思う件数を入れてください（例: 問い合わせ3件 / 面談2件）。
